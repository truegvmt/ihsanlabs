# Architecture Decision Record — v9.1

Ihsan Labs adopts **Next.js App Router as the unified SPA and API layer**, deployed entirely on Vercel, because it eliminates the operational complexity of a separate Express server or Supabase edge function proxy while delivering sub-100ms API routes at the edge with zero cold-start penalty on Vercel's Edge Runtime. The App Router's server components allow the donor flow (intention → resonance preview → allocation) to be server-rendered for first-paint speed without hydrating unnecessary JavaScript, and the co-location of `app/api/route.ts` handlers with page components means the donor flow, IATI metric queries, and allocation requests live in a single deployable unit with shared TypeScript types—removing the cross-service type drift that existed between `index.html` and the Supabase edge functions. The optional PWA is included because Ramadan and high-traffic giving periods frequently occur in regions with intermittent connectivity (Pakistan, Bangladesh, sub-Saharan Africa), and offline-queued donor form submissions materially improve completion rates without adding a separate service worker server—Next.js's `next-pwa` plugin handles this as a build-time asset. Supabase edge functions (`allocation-optimizer`, `due-diligence`, `waqf-agent`, `micro-update-composer`) are retained as the AI layer—they continue to run on Supabase's Deno runtime, invoked by Next.js API routes acting as authenticated proxies, keeping LLM secrets off the Vercel layer and centralizing AI spend under Supabase's observability.
